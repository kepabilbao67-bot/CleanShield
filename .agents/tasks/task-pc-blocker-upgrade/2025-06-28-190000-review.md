# CleanShield: full-stack content blocker with 10 categories, anti-tamper, and AI recovery chatbot

CleanShield went from a minimal hosts-file blocker with three domain lists to a menu-driven Windows protection suite covering 10 harm categories (~1126 domains), DNS-over-HTTPS suppression, browser policy enforcement, scheduled re-application, VPN extension blocklists, NTFS-level hosts protection, and a local HTML/JS chatbot that combines Groq/OpenAI API calls with a 30+ entry offline FAQ aimed at addiction recovery. The batch scripts are entirely rewritten with admin checks, timestamped logging, and error-code tracking. The AI assistant runs from `file://` and falls back gracefully when no API key is configured.

**Watch for:** Password stored in plaintext in the Windows registry (confirmed). The chatbot config file ships with an empty API key but delivers the key over plain HTTP fetch to Groq when configured, and there is no integrity protection on the local HTML/JS files meaning any local user can tamper with the assistant to extract the key or alter its behavior (confirmed). The 30-minute hosts-vigilance task re-runs the full installer silently under SYSTEM, which will spam the log and overwrite any user-made category changes (likely). The batch cleanup regex in `powershell -Command` strips every line matching `^0\.0\.0\.0`, meaning a pre-existing non-CleanShield hosts entry using that prefix will be destroyed on uninstall (confirmed).

## High-level view

The blocking strategy layers hosts-file rewrites with Windows registry browser policies and scheduled tasks. The installer dynamically globs `dominios-*.txt`, so adding a new category is drop-in. DNS-over-HTTPS is disabled at the policy level for Chrome, Edge, Brave, and Firefox, closing the most common bypass for hosts-based blocking. A 30-minute vigilance task and a daily 3 AM re-application task ensure the hosts file is restored if manually edited.

Anti-tamper locks down the hosts file via `icacls` deny rules, blocks browser dev tools, settings pages, extension pages, and known VPN extension IDs. Password protection guards the uninstall path, but the password is stored verbatim in `HKLM\SOFTWARE\CleanShield\Password`; any administrator (or any process reading the registry) can read it.

The AI chatbot is a single-page app loaded from `file://` that calls Groq's OpenAI-compatible endpoint if a key exists in `config.js`, or matches user messages against keyword-scored offline responses. It provides crisis hotlines (024, 112, 717 003 717), addiction recovery guidance, and CleanShield troubleshooting. Because it runs entirely client-side with no server, the API key sits in a plain JS file on disk.

The category configurator saves enabled/disabled state per-category in the registry, then rewrites the hosts file with only the active categories. This interacts badly with the vigilance task: 30 minutes after a user disables a category, the vigilance task re-runs `instalar-bloqueo.bat` which loads all `dominios-*.txt` indiscriminately, restoring the disabled category.

<summary>Issues (7)</summary>

1. **Plaintext password in registry** — `bloquear-desinstalacion.bat` stores the anti-uninstall password as a REG_SZ at `HKLM\SOFTWARE\CleanShield\Password`. Hash it (e.g., `certutil -hashfile` SHA256 comparison) so that reading the registry doesn't reveal the cleartext.
2. **Vigilance task overrides category config** — The 30-minute `CleanShield_VigilarHosts` task calls `instalar-bloqueo.bat` which loads all domain files regardless of the category configuration in the registry. It should call a separate script (or pass a flag) that respects enabled/disabled state stored in `HKLM\SOFTWARE\CleanShield\Categorias`.
3. **Hosts cleanup regex too broad** — The PowerShell cleanup command `Where-Object { $_ -notmatch '^0\.0\.0\.0' }` will delete any pre-existing hosts entries that use `0.0.0.0` as the redirect address, not just CleanShield's. Scope the filter to lines between CleanShield's header/footer markers.
4. **API key exposed in plain JS on disk** — `config.js` contains the Groq API key in cleartext accessible to any local user or program. Consider prompting at runtime and storing in `localStorage` (still imperfect but not a static file), or document clearly that the key grants billable API access.
5. **No input length limit on chatbot messages** — `app.js` sends the full message history (up to 20 messages) plus the new message with no token-counting or size guard. A very long user input could exceed the model's context window and produce a 400 error with no user-facing explanation.
6. **Domain lists include harm-reduction sites** — `dominios-drogas.txt` blocks Erowid and Bluelight, which are harm-reduction resources, not drug marketplaces. For a product aimed at helping people with addictions, blocking harm-reduction information could be counterproductive. Make these configurable or move to a sub-category.
7. **Batch variable expansion bug in configurar-categorias** — The `for /f "tokens=2*"` loop that reads the registry state uses `%%b` for comparison, but REG_DWORD values are reported as hex (`0x0`, `0x1`). The string comparison `if "%%b"=="0"` will never match the actual output format `0x0`, so all categories always appear enabled in the UI.

## Plaintext credential storage

The anti-tamper script prompts for a password and stores it directly:

```batch
reg add "%CLAVE_REG%" /v Password /t REG_SZ /d "!NUEVA_PASS!" /f
```

The uninstall script later reads it back and compares literally:

```batch
if not "!PASS_INPUT!"=="!PASS_GUARDADA!" (
```

Any process running as administrator (or any user with registry read access to HKLM\SOFTWARE) can retrieve the password with `reg query`. For a product positioned as tamper-resistant, this undermines the protection model entirely. A SHA256 hash comparison using `certutil` or PowerShell's `Get-FileHash` equivalent for strings would prevent casual retrieval while keeping the batch-script constraint.

## Vigilance task vs. category configuration conflict

`bloquear-desinstalacion.bat` creates a scheduled task:

```batch
schtasks /create /tn "CleanShield_VigilarHosts" /tr "\"%~dp0instalar-bloqueo.bat\"" /sc minute /mo 30
```

`instalar-bloqueo.bat` unconditionally loads every `dominios-*.txt` file:

```batch
for %%F in ("%SCRIPTDIR%dominios-*.txt") do (
    call :PROCESAR_CATEGORIA "%%F" "%%~nF"
)
```

It never consults the registry state set by `configurar-categorias.bat`. Category customization is effectively wiped every 30 minutes. The fix is either: (a) have the vigilance task call `configurar-categorias.bat` in non-interactive mode with a "reapply current config" flag, or (b) have `instalar-bloqueo.bat` check `HKLM\SOFTWARE\CleanShield\Categorias` before processing each file.

## Hosts file cleanup collateral damage

Three scripts share this cleanup pattern:

```powershell
$c = Get-Content '%HOSTS%' | Where-Object {
    $_ -notmatch 'CleanShield' -and
    $_ -notmatch '^0\.0\.0\.0' -and
    $_ -notmatch '^216\.239\.38\.120'
}
```

Any pre-existing entry like `0.0.0.0 telemetry.microsoft.com` (common in privacy-focused hosts files) will be silently removed. The CleanShield installer already writes header and footer markers (`# === CleanShield - PROTECCION ACTIVA` and `# FIN CleanShield`). The cleanup should remove only lines between those markers rather than globally matching the IP prefix.

## AI chatbot: key exposure and missing size guards

The API key lives in `config.js` as a constant. Because the app runs from `file://`, there is no server-side protection. Any program or user on the machine can read the key. For a product that aggressively locks down browser access and blocks dev tools, leaving an API key in a plain file is an asymmetry worth addressing. At minimum, the key should be entered at runtime and held only in `localStorage`, or `config.js` should be excluded from distribution packaging.

The `callAI` method handles 401 and 429 responses but lacks a guard against oversized requests. With `maxHistoryMessages: 20` and no per-message token counting, a user who pastes large blocks of text repeatedly could exceed the 8K context limit of some models and receive an opaque 400 error with no explanation.

## Registry state comparison bug in category display

In `configurar-categorias.bat`:

```batch
for /f "tokens=2*" %%a in ('reg query "%CLAVE_REG%" /v "%%~nF" 2^>nul ^| findstr /i "%%~nF"') do (
    if "%%b"=="0" set "ESTADO_!NUM!=0"
)
```

`reg query` outputs REG_DWORD values in the format `REG_DWORD    0x0`. With `tokens=2*`, `%%a` gets `REG_DWORD` and `%%b` gets `0x0`. The comparison `"%%b"=="0"` will never match `"0x0"`, so `ESTADO_` is never set to 0. All categories always display as `[X]` (enabled) regardless of their stored state. The fix is to compare against `"0x0"` or parse with `set /a`.

## Domain list editorial choices

The drug category blocks harm-reduction sites like erowid.org and bluelight.org alongside actual marketplaces. These sites provide dosage safety information, drug interaction warnings, and community support that reduces overdose risk. For a product marketed toward addiction recovery, blocking access to "how to use safely if you relapse" information could be counterproductive. Consider splitting into `dominios-drogas-mercados.txt` and `dominios-drogas-informacion.txt` with the latter disabled by default.

## Installer domain-count display may show stale value

The installer uses `set /a TOTAL_DOMINIOS+=1` inside a `:PROCESAR_CATEGORIA` subroutine called from within a `for` loop. The final summary line `Dominios bloqueados: %TOTAL_DOMINIOS%` uses percent-expansion, which cmd.exe resolves at parse time of the enclosing block rather than at execution time. Without `enabledelayedexpansion` at the script level (it's absent from `instalar-bloqueo.bat`), the displayed count will likely show 0 rather than the actual total. The counter still increments correctly; only the display is wrong. Add `setlocal enabledelayedexpansion` at the top and use `!TOTAL_DOMINIOS!` in the summary, or move the summary outside the parsed block.
